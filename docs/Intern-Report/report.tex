This report details the design and development of \textbf{VerbaSense}, a high-performance \Gls{stt} application developed during my internship at \Gls{ussbd}. VerbaSense leverages the OpenAI Whisper engine to provide secure, local, and accurate audio transcriptions.

\subsection{Server Configuration and Resource Management}
Running complex AI models like OpenAI Whisper locally requires a sophisticated approach to server-side resource management. I spent the early weeks of my internship understanding how the local development servers were configured to handle concurrent transcription loads without system degradation.

\begin{itemize}
    \item \textbf{GPU and VRAM Optimization:} Since AI inference is highly dependent on parallel processing, I worked with NVIDIA-based hardware configurations. I experimented with limiting the number of worker threads to prevent CPU thermal throttling, ensuring a consistent performance baseline.
    \item \textbf{Memory-Swapping Behaviors:} I analyzed how the system manages RAM when processing very large audio files (over 100MB). Implementing a stateless data-streaming approach was critical to ensuring that the system did not rely on physical RAM for buffering, which significantly improved the server's stability.
\end{itemize}

\section{Technical Implementation and Development Labs}
This section details the core engineering work and the series of technical experiments conducted to optimize the VerbaSense platform. These "labs" were essential for making data-driven decisions regarding the application's architecture and performance.

\subsection{Core Implementation: OpenAI Whisper and API Layer}
The implementation of the VerbaSense backend required a robust bridge between a modern web environment and a high-performance C++ AI engine. My primary focus was on building a resilient and scalable API layer.

\begin{itemize}
    \item \textbf{Backend Bridge and Process Management:} I utilized the \textbf{Node.js child\_process} module to spawn the Whisper engine as a background worker. This decoupled approach allowed the main Express.js server to remain non-blocking, ensuring that user requests (such as UI updates or history fetching) were handled while the AI engine performed heavy inference in the background.
    \item \textbf{Asynchronous Stream Handling:} I implemented a real-time output parser that captured the engine's stdout. This allowed us to extract the transcribed text and its associated timestamps dynamically, which were then structured into a JSON format for delivery to the frontend.
    \item \textbf{Frontend-Backend Communication:} The communication layer was built using RESTful principles, with a specific focus on handling multi-part form data. I developed custom middleware to manage audio file uploads, ensuring that temporary files were sanitized and deleted immediately after processing to maintain server security and disk integrity.
\end{itemize}

\subsection{Lab 1: Audio Processing and Normalization Benchmarking}
To ensure the highest possible transcription accuracy, I conducted a formal benchmarking lab focused on our \textbf{FFmpeg} pre-processing pipeline.

\begin{itemize}
    \item \textbf{Objective:} To quantify the impact of different audio sample rates and channel configurations on the engine's Word Error Rate (WER) and processing speed.
    \item \textbf{Methodology:} I tested three distinct sets of audio samples: raw 44.1kHz stereo, 16kHz mono, and 8kHz telephone-quality mono. Each sample set was passed through the Whisper engine, and the transcription accuracy was manually verified against a ground-truth transcript.
    \item \textbf{Observations:} While 44.1kHz audio offered high fidelity, it significantly increased processing time without a proportional increase in accuracy. Conversely, 8kHz audio led to numerous "hallucinations" in the AI output. The 16kHz mono format provided the optimal "sweet spot" for accuracy and inference speed.
    \item \textbf{Outcome:} I implemented a mandatory pre-processing step using FFmpeg that automatically re-samples and normalizes every incoming file to 16kHz mono. This standardization significantly improved the reliability of the VerbaSense platform across diverse user recordings.
\end{itemize}

\subsection{Lab 2: Inference Speed and Memory Optimization}
This lab focused on the critical task of balancing model complexity with the hardware limitations of our local development servers.

\begin{itemize}
    \item \textbf{Objective:} To identify the optimal OpenAI Whisper model weight (Tiny, Base, or Small) for production-grade local deployment without exceeding system VRAM limits.
    \item \textbf{Methodology:} I executed a series of stress tests using each model variant on our workstation’s NVIDIA GPU. I monitored VRAM consumption using \textit{nvidia-smi} and recorded the "Real-Time Factor" (RTF)—the ratio of processing time to audio duration—for each test case.
    \item \textbf{Observations:} The "Tiny" model was exceptionally fast but struggled with complex accents. The "Small" model provided excellent accuracy but consumed nearly 80\% of the available VRAM, which would limit system concurrency. The "Base" model offered high accuracy (nearly 95\% of the "Small" model's quality) while maintaining a low VRAM footprint.
    \item \textbf{Outcome:} Based on these data-driven findings, my supervisor and I decided to implement the "Base" model as our production default. This decision ensured that the platform could handle multiple simultaneous users while maintaining a professional standard of transcription
\end{itemize}

\section{Tools and Technologies Used}
The development of the VerbaSense platform relied on a modern, robust, and highly integrated tech stack. This section details the tools and technologies selected to meet the project's intensive computational and security requirements, as specified in the official project documentation.

\subsection{Frontend Development Stack}
To deliver a responsive and high-performance user experience, the frontend was built using the \textbf{Next.js} framework. This choice allowed for efficient server-side rendering and optimized page loads.
\begin{itemize}
    \item \textbf{TailwindCSS and Shadcn UI:} For styling, I utilized TailwindCSS for its utility-first approach, combined with Shadcn UI for accessible and aesthetically pleasing components.
    \item \textbf{Framer Motion:} To enhance user engagement, I implemented smooth UI transitions and micro-animations using Framer Motion, giving the platform a premium feel.
    \item \textbf{Web Audio API:} I integrated the Web Audio API to enable direct, high-fidelity microphone recording within the browser, facilitating a seamless user-to-server audio pipeline.
\end{itemize}

\subsection{Backend and Security Framework}
The backend was architected using \textbf{Node.js} and \textbf{Express.js}, implemented with \textbf{TypeScript} to ensure type safety and long-term maintainability.
\begin{itemize}
    \item \textbf{Authentication and Security:} User security was prioritized through the implementation of \textbf{JSON Web Tokens (JWT)} for session management and \textbf{Bcrypt} for robust password hashing.
    \item \textbf{Audio Stream Management:} I used the \textbf{Multer} middleware to handle the secure upload and temporary storage of `.wav` files, ensuring that the backend remained efficient even during high-concurrency transcription tasks.
\end{itemize}

\subsection{AI Inference and Audio Engine}
The core of the application is the \textbf{OpenAI Whisper} engine, which was integrated to provide industry-leading speech-to-text accuracy.
\begin{itemize}
    \item \textbf{Whisper.cpp and FFmpeg:} I utilized the C++ wrapper for Whisper to enable local, high-performance inference. \textbf{FFmpeg} served as the essential audio "sanitizer," normalizing input files to the 16kHz mono format required by the AI model.
    \item \textbf{Docker Isolation:} The inference engine was containerized using \textbf{Docker}, ensuring that the AI environment remained isolated from the main application server, preventing resource contention and improving system stability.
\end{itemize}

\subsection{Database and Cloud Integrations}
The platform leverages a hybrid storage strategy to balance performance with data sovereignty.
\begin{itemize}
    \item \textbf{MongoDB and Mongoose:} All user metadata and transcription histories are stored in \textbf{MongoDB}, with Mongoose providing a structured object-modeling layer.
    \item \textbf{Cloudinary and Brevo:} I integrated \textbf{Cloudinary} for the permanent storage of user profile images and voice metadata. For automated communication, I utilized the \textbf{Brevo REST API} to handle OTP (One-Time Password) verification and account security notifications.
\end{itemize}

\subsection{DevOps and Deployment Pipeline}
A professional CI/CD pipeline was established to maintain high project velocity and deployment reliability.
\begin{itemize}
    \item \textbf{GitHub and Vercel:} \textbf{GitHub} served as the central version control hub, while \textbf{Vercel} was used for high-speed frontend hosting.
    \item \textbf{Render and Docker:} The backend services were deployed on \textbf{Render}, utilizing Docker containers to ensure that the complex environment required for AI inference remained consistent across staging and production environments.
\end{itemize}
