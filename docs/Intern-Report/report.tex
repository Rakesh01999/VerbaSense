\chapter*{Abstract}
\addcontentsline{toc}{chapter}{Abstract}
This report shows a detailed overview of the design and development of \textbf{VerbaSense}. Which is a high-performance Speech to-Text (STT) application developed during my internship at Unicorn Software \& Solutions Ltd (USSBD). VerbaSense leverages the OpenAI Whisper engine to provide secure, local and accurate audio transcriptions.

The project was built using a modern full-stack architecture. That includes Next.js 14 for the frontend and Node.js/Express for the backend. Key features include : real-time transcription, history management and secure JSON Web Token (JWT)-based authentication. This report covers the methodology, system architecture, implementation details and evaluation of the platform.

\subsection{Server Configuration and Resource Management}
Running AI models like Whisper on a local computer needs a lot of careful planning. In the first few weeks, I worked closely with my supervisor to learn how our servers were set up to handle multiple transcriptions at once without slowing down or crashing.

\begin{itemize}
    \item \textbf{GPU and Memory (VRAM):} Since AI works best with graphics cards (GPUs), I learned how to monitor the graphics memory. This was important to make sure we weren't trying to run too many transcription tasks at once, which could run out of memory and stop the service.
    \item \textbf{Processing Power (CPU):} For tasks like cleaning up and fixing audio files before they are transcribed, the main processor (CPU) does a lot of work. I learned how to manage these tasks so the computer didn't get too hot or slow down other important office work.
    \item \textbf{Handling Large Files:} When someone uploads a very large audio file, it can easily use up all the available memory. I learned how to process the data in small pieces (streaming) so the system stayed stable even when dealing with very long recordings.
\end{itemize}

\subsection{Development Network and Secure Access}
Data security and user privacy are paramount concerns at \Gls{ussbd}. The development network was logically and physically segmented to provide a secure environment for testing the VerbaSense platform.

\begin{itemize}
    \item \textbf{Dedicated Research Workspace:} I worked in a dedicated research setup that was kept separate from the rest of the office's main systems. This was very important because testing heavy AI models and processing large files needs a lot of power, and this separation made sure my work didn't slow down or disturb other employees in the branch.
    \item \textbf{Secure Data Handling Protocols:} I learned to implement encrypted communication channels (HTTPS) for all interactions between the web server, the transcription engine and the MongoDB instances. This taught me the importance of securing "data in transit."
    \item \textbf{Access Control and Authentication:} Following the "Least Privilege" principle, I was granted access only to the specific servers and databases required for the project. This exposure to professional access control provided me with a solid understanding of how \Gls{ussbd} protects its intellectual property and user data.
\end{itemize}

\section{Technical Implementation and Development Labs}
This section details the core engineering work and the series of technical experiments conducted to optimize the VerbaSense platform. These "labs" were essential for making data-driven decisions regarding the application's architecture and performance.

\subsection{Core Implementation: OpenAI Whisper and API Layer}
The implementation of the VerbaSense backend required a robust bridge between a modern web environment and a high-performance C++ AI engine. My primary focus was on building a resilient and scalable API layer.

\begin{itemize}
    \item \textbf{Backend Bridge and Process Management:} I utilized the \textbf{Node.js child\_process} module to spawn the Whisper engine as a background worker. This decoupled approach allowed the main Express.js server to remain non-blocking, ensuring that user requests (such as UI updates or history fetching) were handled while the AI engine performed heavy inference in the background.
    \item \textbf{Asynchronous Stream Handling:} I implemented a real-time output parser that captured the engine's stdout. This allowed us to extract the transcribed text and its associated timestamps dynamically, which were then structured into a JSON format for delivery to the frontend.
    \item \textbf{Frontend-Backend Communication:} The communication layer was built using RESTful principles, with a specific focus on handling multi-part form data. I developed custom middleware to manage audio file uploads, ensuring that temporary files were sanitized and deleted immediately after processing to maintain server security and disk integrity.
\end{itemize}

\subsection{Lab 1: Audio Processing and Normalization Benchmarking}
In this lab, I focused on making sure the audio files were in the best possible format before sending them to the transcription engine. I tested different types of audio—like high-quality music files, standard mono recordings and low-quality phone recordings. I noticed that very high-quality files took too much time to process without actually making the text any better, while low-quality files caused the AI to make a lot of mistakes. After some testing, I found that converting everything to a 16kHz mono format using FFmpeg was the perfect balance. It made the transcriptions much faster and more reliable, regardless of how the user recorded their audio.

\subsection{Lab 2: Inference Speed and Memory Optimization}
For the second lab, I needed to figure out which AI model would work best on our office computers. I tested the Tiny, Base and Small versions of the Whisper model to see how they handled speed and computer memory. The Tiny model was very fast but not accurate enough and the Small model was accurate but used up almost all the computer's memory, which would slow everything else down. I found that the Base model was the best choice—it was accurate enough for professional use but didn't crash the system. Because of this, my supervisor and I decided to use the Base model for VerbaSense, ensuring it runs smoothly for everyone.

\printchaptertitle{Technical Challenges and Solutions}
The development of \textbf{VerbaSense} was not without significant technical hurdles. Building a local AI-powered platform required solving complex problems related to hardware limitations, data security and high-concurrency processing. This chapter details the primary challenges encountered during the internship and the engineering strategies implemented to overcome them.
\section{Challenges Faced and Solutions}
Throughout the project lifecycle, I encountered several "bottlenecks" that required a combination of research, experimentation and collaborative problem-solving with my supervisor.
\subsection{Large File Upload Handling}
\textbf{The Challenge:} Early in the development phase, the Express.js backend frequently crashed when users attempted to upload audio files larger than 10MB. The default memory limits of Node.js were insufficient to handle multiple simultaneous large-file uploads, leading to "Heap Out of Memory" errors.
\textbf{The Solution:} I solved this by implementing a two-pronged approach. First, I adjusted the Express payload limits and configured \textbf{Multer} to use disk-based storage instead of memory-based buffering. This ensured that large files were streamed directly to the server's temporary storage, bypassing the RAM bottleneck. Second, I implemented a client-side validation layer that warned users about file size limits before the upload process began, significantly improving the system's stability.
\subsection{Model Inference Latency}
\textbf{The Challenge:} Initial tests using the Whisper "Large" model on standard CPU hardware resulted in unacceptable latency—sometimes taking up to five times the duration of the audio itself to produce a transcript. For a production-ready application, this was a significant barrier to user adoption.
\textbf{The Solution:} After benchmarking different configurations in our technical labs, I moved the inference engine to the \textbf{graphics card} using the C++ optimized version of Whisper. I also implemented a dynamic model-selection strategy where the "Base" model was used for quick previews, while the more intensive models were reserved for high-accuracy requests. This optimization reduced transcription time by over 70\%, making the platform feel significantly more responsive.
\subsection{Audio Format Incompatibility}
\textbf{The Challenge:} The OpenAI Whisper engine is highly sensitive to input formats, requiring specific 16kHz mono WAV files. However, users frequently uploaded a wide variety of formats, including MP3, FLAC and high-bitrate stereo WAVs, which often caused the AI engine to fail or produce distorted transcripts.
\textbf{The Solution:} I integrated a robust audio pre-processing pipeline using \textbf{FFmpeg}. I developed an automated script that intercepted every uploaded file and performed a "sanitization" pass—automatically converting it to the required 16kHz mono PCM format. This ensured that the AI engine received a consistent, high-quality stream every time, regardless of the user's original recording quality or format.
\subsection{Secure Session Management}
\textbf{The Challenge:} Because VerbaSense handles private and sensitive audio data, ensuring secure session management was paramount. Traditional local-storage-based authentication was deemed too risky due to potential XSS (Cross-Site Scripting) vulnerabilities.
\textbf{The Solution:} I implemented a robust security architecture using \textbf{JSON Web Tokens (JWT)} stored exclusively in \textbf{HTTP-only cookies}. This approach prevents client-side scripts from accessing the session token, effectively mitigating the risk of token theft. Additionally, I configured a strict \textbf{CORS (Cross-Origin Resource Sharing)} policy to ensure that only authorized frontend domains could interact with our transcription API.
\subsection{Resource Optimization for Local Deployment}
\textbf{The Challenge:} During peak usage simulations, multiple concurrent transcription requests would often "max out" the server's CPU and VRAM, causing the entire system to hang or drop active connections.
\textbf{The Solution:} I implemented a sophisticated \textbf{Task Queue} system using a producer-consumer pattern. Instead of processing every request immediately, the system now queues incoming tasks in a MongoDB-backed manager. A specialized "Worker" process then picks up these tasks sequentially based on the available system resources. This prevents system overloads and ensures that even during high-traffic periods, every user eventually receives their transcript with 100\% accuracy.
